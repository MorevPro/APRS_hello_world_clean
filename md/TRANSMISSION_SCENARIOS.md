## ❌ Сценарий 1: Буфер переполнен (Buffer Overflow)

```
❌ ПРОБЛЕМА: APRS_HELLO_WORLD_TX_LEVEL_CAPACITY = 2048 (слишком мало?)

@aprs_refresh_frame():
  Frame size: 65 bytes
  Expected NRZI bits: ~850 bits
  But actual NRZI bits reported: 2048 bits ← ПОЛНЫЙ БУФЕР!

ПРИЧИНА: 
  Bit stuffing добавляет слишком много битов, или
  Что-то в коде не считает правильно...

РЕЗУЛЬТАТ:
  app->tx_level_count = 2048  ← обрезано!
  Но должно быть: 850
  
  TX передает только первые N бит:
  [Пребабула] [90% фрейма] [ничего постабулы!]
                            ↑ ФРЕЙМ РЕЖЕТСЯ ЗДЕСЬ!

📻 На рации слышно:
  Первая часть звука правильная, потом просто тишина
  (потому что постабула обрезана)

📡 Direwolf видит:
  Неполный пакет → не может декодировать → ничего не показывает

РЕШЕНИЕ:
  Увеличить APRS_HELLO_WORLD_TX_LEVEL_CAPACITY до 4096
  #define APRS_HELLO_WORLD_TX_LEVEL_CAPACITY 4096U
```

## ❌ Сценарий 2: NRZI кодирование неправильное

```
❌ ПРОБЛЕМА: Bitstuffing или уровни инвертированы

Правильное NRZI кодирование флага 0x7E:
  Бит-по-бит:   0 1 1 1 1 1 1 0  (LSB first)
  NRZI уровни: 1 0 1 0 1 0 1 0  (toggle на каждый 0-бит)
  
  Ожидаемый вывод: "01010101"

Неправильное кодирование (инвертированные уровни):
  Выводит: "10101010"  ← Противоположено!
  
  Или: "11111100"  ← Нет toggling! (потеря bit stuffing)

РЕЗУЛЬТАТ:
  Direwolf пытается декодировать, но не находит синхронизации
  (т.к. флаги 0x7E потеряны или неправильные)

  Вывод в Direwolf:
  (ничего - молчит, т.к. не нашел синхро)

ДИАГНОСТИКА:
  @aprs_refresh_frame(), проверьте вывод:
  "First 100 NRZI bits (True=1, False=0):"
  
  ✓ ПРАВИЛЬНО: "0101010101010101110110110..."  ← много чередований
  ✗ НЕПРАВИЛЬНО: "1111111111110000000..." ← сплошные блоки

РЕШЕНИЕ:
  Проверить функцию aprs_ax25_emit_byte_nrzi():
  
  static void aprs_ax25_emit_byte_nrzi(...) {
      for(uint8_t bit_index = 0; bit_index < 8U; bit_index++) {
          const bool bit = ((value >> bit_index) & 0x01U) != 0U;

          if(!bit) {
              *current_level = !(*current_level);  ← ВАЖНО: toggle на 0!
              *consecutive_ones = 0U;
          } else if(bit_stuffing) {
              (*consecutive_ones)++;
          }
          
          // Вывести текущий уровень
          aprs_ax25_emit_bit(*current_level, ...);
      }
  }
  
  Если логика здесь неправильная → весь NRZI сломан!
```

## ❌ Сценарий 3: TX Callback прерван раньше времени

```
❌ ПРОБЛЕМА: Передача останавливается в середине пакета

Диагностика в aprs_radio_poll():
  
  TX finished successfully
    Bits sent: 452 / 987  ← ТОЛЬКО ПОЛОВИНА!
    Missing bits: 535

ПОЧЕМУ ЭТО ПРОИСХОДИТ:
  
  1. Таймер истек раньше, чем нужно
  2. CC1101 модуль сбросился
  3. Flipper перешла в режим сна/питания
  4. Переполнение очереди событий

РЕЗУЛЬТАТ:
  app->tx_running = false  (TX callback больше не вызывается)
  furi_hal_subghz is_async_tx_complete() = true
  
  TX прерывается, но должен был отправить еще 535 бит (~55ms)!

  На рации слышно:
  [Первая половина нормального звука] [РЕЗКО ОБРЫВАЕТСЯ]

  Direwolf видит:
  Начало пакета (пребабула и часть данных) → потом молчание
  → Не может синхронизироваться на половине пакета → игнорирует

ДИАГНОСТИКА:
  Проверьте в логах:
  
  TX_DEBUG: freq=432500000 Hz, bits=987, duration=102 ms ← ОК
  TX_DONE: sent=452 of 987 bits, INCOMPLETE! ← ОШИБКА!

РЕШЕНИЕ:
  1. Проверить, не есть ли таймер с коротким timeout
  2. Убедиться, что питание включено на время TX
  3. Может быть bug в furi_hal_subghz или переполнение очереди
```

## ❌ Сценарий 4: Неправильные координаты → Fallback на Status Frame

```
❌ ПРОБЛЕМА: Position рассчитывается как INVALID

@aprs_refresh_frame():
  
  Position is INVALID, encoding status frame

ПРИЧИНА:
  aprs_ax25_position_is_valid() возвращает false
  
  Возможные причины:
  1. Position = "00.0000N 000.0000E" (default/zero)
      → Явно отбросит как invalid
  2. Position не содержит N/S/E/W (неправильный формат)
  3. Position пустая (NULL или пустая строка)

РЕЗУЛЬТАТ:
  Вместо Position Frame → Status Frame
  Status Frame меньше (~45 байт vs 65)
  
  Direwolf видит:
  [0.5] KV2TST-0>APRS-0,WIDE1-1:>Flipper Zero APRS Test
  (No position in this packet!)

РЕШЕНИЕ:
  Проверьте в Settings APRS приложения:
  - Latitude: Должна быть в формате "5549.50N" или "55 49.50N"
  - Longitude: Должна быть в формате "03737.49E" или "037 37.49E"
  
  НЕПРАВИЛЬНЫЕ форматы:
  ✗ "55.8241" (просто число, нет N/S)
  ✗ "55 82 41" (разделитель пробелом без дектора)
  
  ПРАВИЛЬНЫЕ форматы:
  ✓ "5549.50N" (DDMM.SSN - обязательно N/S!)
  ✓ "03737.49E" (DDDMM.SSE - обязательно E/W!)
```

## 📐 Вычисление ожидаемого размера биттов

```c
Формула расчета:

NRZI_bits_before_stuffing = (preamble_flags + frame_size + postamble_flags) × 8

Для типичного Position Frame:

Preamble flags:   32 × 8  = 256 bits
Frame size:       65 × 8  = 520 bits

Bit-stuffing добавляет ~0 целый 0-бит после каждых 5 подряд идущих 1-ов
В AX.25 position frame обычно ~15-20% bit-stuffing = +5-10%

Stuffed bits:     520 × 0.08 = ~42 bits

Postamble flags:  3 × 8    = 24 bits

ИТОГО: 256 + 520 + 42 + 24 = 842 bits ≈ 850-900 bits

Это ЛЕГКО умещается в 2048 bit буфер!

Если видите 2048 бит → ПЕРЕПОЛНЕНИЕ, что-то не так!
Если видите 0 бит → NRZI кодирование не произошло!
Если видите 800-900 бит → ВСЕ ОК! ✓
```

## 🎯 Контрольный список диагностики

Просмотрите логи по этому чек-листу:

```
[ ] Frame size между 45-100 байт?
    ├─ Если 0 → Position/Status frame кодирование не произошло
    └─ Если > 200 → Что-то добавляет лишние данные

[ ] NRZI bits между 500-1500?
    ├─ Если 0 → aprs_ax25_build_nrzi_stream() не вызвалась
    ├─ Если 2048 → Буфер переполнен (увеличить CAPACITY)
    └─ Если 800-900 → Нормально! ✓

[ ] Первые 100 NRZI бит содержат "0101ххххх"?
    ├─ Если "11111111" → Bit stuffing или уровни неправильные
    ├─ Если "00000000" → На всех NRZI битах одна логика
    └─ Если чередующиеся → Нормально! ✓

[ ] TX started успешно?
    ├─ Если нет → furi_hal_subghz_start_async_tx() вернула false
    └─ Если да → Передача началась ✓

[ ] TX complete с полным количеством бит?
    ├─ Если bits_sent < bits_expected → TX прерван раньше
    ├─ Если bits_sent == bits_expected → Все OK! ✓
    └─ Проверьте что missing_bits = 0

[ ] Direwolf получил сигнал?
    ├─ Если "Audio level 0%" → Микрофон ничего не слышит
    ├─ Если "Audio level 50%" → Сигнал есть!
    ├─ Если декодировано сообщение → УСПЕХ! 🎯
    └─ Если ничего не декодировано → Сигнал испорчен где-то
```

## 🚀 Наиболее вероятные причины в вашем случае

На основе вашего описания ("слышит звуки разной тональности" но не декодирует):

```
Вероятность 1️⃣ (60%): БУФЕР ПЕРЕПОЛНЕН
  → Биты обрезаны после 2048 символов
  → Direwolf слышит неполный пакет
  → Не может синхронизироваться
  → Не декодирует

Вероятность 2️⃣ (25%): NRZI КОДИРОВАНИЕ НЕПРАВИЛЬНОЕ
  → Bit stuffing считается неправильно
  → Уровни (True/False) инвертированы
  → Direwolf не находит синхронизирующих флагов 0x7E
  → Не декодирует

Вероятность 3️⃣ (10%): TX ПРЕРЫВАЕТСЯ РАНЬШЕ ВРЕМЕНИ
  → Таймер или переполнение очереди
  → Вторая половина пакета не отправляется
  → Неполный пакет
  → Не декодирует

Вероятность 4️⃣ (5%): КООРДИНАТЫ НЕПРАВИЛЬНЫЕ
  → Fallback на Status Frame (меньше данных)
  → Может быть это и работает, просто не видно положения
```
