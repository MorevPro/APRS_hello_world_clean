Проверка на устройстве прошла успешно: на `2026-04-02` Flipper уже работает на `Unleashed` `unlshd-086`, `device_info` показывает `hardware_region_provisioned : 00`, и тестовая CLI-команда

```text
subghz tx 000000 432500000 100 3 0
```

отработала без ошибки `Transmission on this frequency is restricted in your region`. Значит, передавать на `432500000` теперь можно.

Приложение я пересобрал и заново запустил на Flipper. В текущем состоянии передача в приложении устроена так:

- `Right` переключает режим `RX` / `LABTX`
- `Left` включает или выключает arm для передачи
- удержание `OK` запускает передачу
- `Up` пересобирает APRS-кадр
- `Down` сбрасывает RX counters
- `Back` выходит из приложения

Чтобы отправить данные APRS через радио канал, делайте так:

1. Запустите приложение `APRS Cln Hello World application`.
2. Убедитесь, что в верхней строке экрана режим стал `LABTX`, если нет, нажмите `Right`.
3. Нажмите `Left`, чтобы стало `Arm:yes`.
4. Удерживайте центральную кнопку `OK`.
5. В этот момент приложение запускает передачу тестового beacon-пакета на `432.500 MHz`.

На экране полезно смотреть на это:
- строка `APRS 432.500 LABTX` — мы точно в режиме передачи
- строка `Arm:yes TX:busy 00` — передача реально запущена, регион открыт
- `Frame:... Bits:...` — кадр собран
- `last_status` внизу — увидите `TX in progress` и потом `TX complete`
- строка с hex preview — это первые байты собранного AX.25 кадра

Для терминальной отладки это тоже удобно. Я добавил в приложение логи, поэтому можно открыть CLI и смотреть их так:

```powershell
ufbt cli
```

потом в CLI Flipper:

```text
log info
```

После этого на самом Flipper руками нажимайте кнопки. В логе стоит ожидать такие сообщения:
- `Radio init: region=00 freq=432500000 tx_allowed=yes`
- `RX started at 432500000 Hz`
- `Mode changed to LAB TX`
- `TX armed=yes`
- `TX started: freq=432500000 frame=... bits=... preview=...`
- `TX complete: packets=... last_rssi=... last_lqi=...`

Если вместо этого появится что-то вроде:
- `TX request rejected: current mode is RX`
- `TX request rejected: tx not armed`
- `TX start failed`

то это уже точная причина, где ломается сценарий.

На что смотреть для дальнейшей отладки beacon:
- есть ли `TX started` и `TX complete` в логах;
- меняется ли на экране `TX:idle` на `TX:busy`;
- есть ли вспышка красного LED во время передачи;
- видит ли второй приёмник сам факт несущей на `432.500`;
- если проверяете Баофенгом, помните: это не голосовой FM-сигнал, а цифровой `Sub-GHz/GFSK` тестовый поток, поэтому “нормальной APRS-голосовой модуляции” вы там не услышите.

Очень важный момент: сейчас приложение шлёт лабораторный beacon, а не финальный валидный APRS beacon для эфира. В коде пока зашит источник `MYCALL-1` и статусный текст формируется автоматически в [aprs_hello_world_clean_app.c](D:/projects/HAM/flipper%20zero/APRS_hello_world_clean/aprs_hello_world_clean_app.c) и [aprs_ax25_encoder.c](D:/projects/HAM/flipper%20zero/APRS_hello_world_clean/aprs_ax25_encoder.c). То есть следующий шаг для “корректно передавать beacon aprs” — проверить совместимость сигнала на принимающей стороне и заменить тестовые поля на реальные APRS-параметры.
